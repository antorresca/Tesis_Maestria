### Cambios realizados

**1. `modules/pln/src/dataset.py`**

Agregada augmentación de texto **online** (en `__getitem__`, no genera CSV nuevo):

- `_augment_text()`: aplica ruido aleatorio en cada época → el modelo ve variaciones distintas cada vez
- **Random word deletion** (`word_del_p=0.05`): elimina palabras con 5% de probabilidad por palabra (nunca elimina todas). Simula comandos parciales del usuario.
- **Character-level swap** (`char_swap_p=0.02`): intercambia pares de caracteres adyacentes con 2% de probabilidad. Simula typos reales.
- Argumento `augment: bool = False` en `__init__`: se activa **solo en train**, nunca en val/test (métricas honestas).

**2. `modules/pln/src/train.py`**

- Agregado flag `--augment` para activar augmentación online en el train dataset.
- Agregado `label_smoothing_factor=0.05` en `TrainingArguments`: impide que el modelo alcance probabilidad 1.0 en una clase, frenando la sobreconfianza que producía `train_f1=1.0`.

> **Nota de calibración:** Se probó primero con `word_del_p=0.10`, `char_swap_p=0.05`, `label_smoothing=0.10`. Los resultados (86.4%) fueron mejores que el punto de partida (84.8%) pero inferiores al 90.9% histórico. Análisis: los parámetros altos son destructivos en frases cortas de 3-5 palabras (p.ej. borrar el verbo de "ve al pasillo"). Se redujeron a los valores actuales.

### Decisión de diseño: reducción a 3 clases atómicas

**Motivación:** Los dos runs con augmentación dieron consistentemente 86.4% F1. El análisis de las matrices de confusión reveló que los errores no son aleatorios — son semánticamente coherentes:

- `go_home → navigate` (2 errores): go_home ES un navigate con `destination="home"`
- `pick → fetch` (2 errores): fetch ES un navigate + pick
- `transport` y `fetch` son acciones compuestas, no atómicas

Agregar más datos para discriminar pares que son semánticamente idénticos es luchar contra la física del problema. La solución arquitectónica correcta es que PLN solo clasifique acciones verdaderamente atómicas.

**Nueva ontología de intenciones PLN (3 clases):**

| Clase PLN | Reemplaza a | Entity extractor captura |
|---|---|---|
| `navigate` | `navigate` + `go_home` | `destination` (`"home"` hardcoded para go_home) |
| `pick` | `pick` + `fetch` | `target` (+ `destination` si fetch) |
| `place` | `place` + `transport` | `destination` (+ `target` si transport) |

**Remapeo de etiquetas en dataset:**

| Original | → Nueva |
|---|---|
| navigate | navigate |
| go_home | navigate |
| pick | pick |
| fetch | pick |
| place | place |
| transport | place |

**Resolución del naming collision PLN vs Task Planning:**

PLN y Task Planning usan `pick` y `place` con semántica distinta. Convención decidida:

- **PLN output:** campo `intent` con valores `navigate | pick | place` — representa la _intención de alto nivel_ del usuario
- **Task Planning interno:** usa "subtareas" con verbos operacionales (`move_to`, `grasp`, `release`) — nunca reutiliza los nombres de intención PLN

Interfaz PLN → Task Planning:
```json
{ "intent": "pick", "target": "botella", "destination": "mesa_B", "confidence": 0.95 }
```

Task Planning interpreta: pick + destination presente → secuencia interna [move_to(target), grasp(target), move_to(destination), release()]

**Task Planning:** se implementará como árbol de decisiones jerárquico. HRL es opcional según tiempo disponible.

**Pipeline ejecutada:**

1. Remapeo en `prepare_dataset.py` (sin tocar CSVs raw) — `INTENT_REMAP` dict + `df.replace()` antes del clean
2. `python src/prepare_dataset.py` → 436 ejemplos, 3 clases balanceadas (~145/clase), splits 305/65/66
3. `python src/train.py --model mbert --unfreeze_layers 0 --augment`
4. Corregido `evaluate.py`: `INTENTS` hardcodeado → lee `id2label` del `config.json` de cada modelo
5. `python src/evaluate.py`
