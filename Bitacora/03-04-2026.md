### Ronda 1 — Fix timing reset (reset_world sin pausa)

**Problema:** La auto-colisión persistía en los resets de episodio. El flujo anterior era:

```
reset_world → sleep(0.8 s) → leer estado → enviar goal home × 10
```

Durante esos 0.8 s el WBC corría con el **goal del episodio anterior**. El robot regresaba al spawn (joints=0) pero el WBC aplicaba torques para alcanzar ese goal viejo → contorsión del brazo → auto-colisión.

**Fix:** Se añadió parámetro `ee_home` a `robot_interface.reset()`. Inmediatamente tras `reset_world` (sin sleep) se envían 5 mensajes con el home goal antes de `time.sleep(settle_time)`.

```
reset_world → send_goal(home) × 5 → sleep(0.8 s) → send_goal(real) × 10
```

**Fix 2 relacionado:** Los training goals de navigate tenían orientaciones grandes (qz=1.0, qw=0.0 → 180°). El SLERP interpolaba lentamente hacia esa orientación durante el episodio, contorsionando el brazo. Todos los goals navigate cambiados a orientación identidad (qw=1). Eliminado el goal `x = _EE_HOME − 0.6` (detrás del robot, inalcanzable en exploración temprana).

**Fix 3 relacionado:** `mobile_manipulator_env.reset()` ahora pasa `ee_home=_EE_HOME` al `ri.reset()`. `_EE_HOME` importado de `config.py`.

### Ronda 2 — Problemas reportados tras smoke test

**Problema 1 — train.py no inicia la simulación:**
`init_goal_publisher.py` publica el hold-pose a 50 Hz durante 5 s tras el primer `/data` del WBC. Si `model.learn()` empieza mientras el publisher sigue activo, ambos compiten en `/mobile_manipulator/desired_traj`. El publisher gana por frecuencia (50 Hz vs ~10 Hz del DRL) → el robot no responde a las acciones DRL → "la simulación no inicia".

**Problema 2 — Movimientos erráticos y auto-colisión durante exploración:**
Durante la fase de exploración aleatoria, el DRL puede tomar ACTION_MOVE_DOWN repetidamente. Con `WORKSPACE_Z[0] = 0.05 m`, el EE puede bajar hasta casi el suelo en 23 steps `(0.747 − 0.05) / 0.03 ≈ 23`. El brazo en esa configuración toca la base → auto-colisión.

**Problema 3 — Reset poco robusto para cuando la simulación "se pierde":**
Cuando Gazebo entra en estado explosivo (robot volando/NaN), `reset_world` sin pausar la física puede dejar al WBC con torques erráticos durante el periodo de settle.

### Ronda 2 — Fixes aplicados

#### `robot_interface.reset()` — pause/unpause alrededor del reset

Secuencia definitiva:
```
pause_physics
  → reset_world          (physics frozen: robot vuelve a spawn, sin torques)
  → send_goal(home) × 10 (WBC encola home goal con physics pausada)
unpause_physics           (WBC aplica torques hacia home desde el ciclo 1)
  → sleep(settle_time)   (WBC converge a home)
  → read state
  → send_goal(real) × 10 (flush final con orientación real RPY→quat)
```

La física permanece **congelada** mientras el WBC recibe el home goal. Al reanudar, el primer torque ya apunta a home. Elimina completamente la ventana de torques erróneos.

Nuevas constantes de clase: `_SVC_PAUSE = '/gazebo/pause_physics'`, `_SVC_UNPAUSE = '/gazebo/unpause_physics'`.

#### `config.py` — WORKSPACE_Z mínimo: 0.05 m → 0.35 m

Para entrenamiento navigate: EE home = 0.747 m → solo puede bajar 0.397 m antes del límite. Previene brazo contorsionado durante exploración aleatoria.
**Nota:** Bajar a 0.10 m cuando se entre a entrenamiento de pick/place (EE necesita llegar a z ≈ 0.15 m).

#### `train.py` — Wait de inicialización antes de model.learn()

```python
_INIT_WAIT = 6.0  # s
time.sleep(_INIT_WAIT)
```
El env ya esperó el primer estado (WBC activo). El sleep cubre los ~5 s restantes de `init_goal_publisher` + margen.
